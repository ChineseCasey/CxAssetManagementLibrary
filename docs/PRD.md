## 1. 背景与问题陈述

### 1.1 当前状态（基于现有代码）

- **形态**：以 PySide2/PySide 为主的桌面端资产浏览（如 `CXA_Script/CXMainWin.py`），依赖本机 UI 与脚本环境，协同与部署成本高。
- **数据层问题（核心）**：当前以 **`CXA_Library` 目录 + `CXDataFile.py` 的嵌套 `dict`** 为「全量数据模型」——在原型阶段可跑，但**不是产品化模型**：
  - 难约束不变量（主文件、版本、标签、软删除、冲突解决）；
  - 难做多条件检索、统计、审计与增量更新；
  - 与「未来 manifest / 管线回写 / 多库多租户」难以演进。
- **工程可维护性**：部分代码风格偏旧、错误处理与可观测性不足，与 Web、并发、权限等需求叠加时风险高。

### 1.2 为何需要「数据库」而不仅是「扫一遍目录」

| 能力 | 仅依赖实时目录遍历/内存 dict | 数据库 + 受控同步 |
|------|------------------------------|------------------|
| 元数据与业务字段扩展 | 改结构即伤全局，难迁移 | 通过迁移脚本版本化 |
| 搜索与排序 | 大库时反复 walk，延迟不可控 | 可建索引、分页稳定 |
| 与磁盘不一致 | 难表达「已扫描/待校验/已损坏」等状态 | 用状态机与任务表可表达 |
| 多进程/多实例 | 几乎无法安全共享 | 连接池 + 事务 |
| 未来写操作/协作 | 极难 | 有事务与约束基础 |

**结论（PRD 立场）**：本重构的 **V1 起即以关系型（或等效可迁移的 SQL 抽象）为目录与元数据的「权威存储」之一**；磁盘上的 `CXA_Library` 作为 **ingest/同步源（Source of files）**，**不是**长远的唯一真源。若与磁盘冲突，**以可配置策略为准**（V1 默认：只读同步，不擅自改盘）。

### 1.3 要解决的问题（不变，强调顺序）

1. **跨端访问**：浏览器中浏览、检索、预览。  
2. **健壮 + 可演进**：以数据库与同步管道支撑产品级模型。  
3. **体验**：现代化、美观的 Web UI。  
4. **后端 Python**：与管线/DCC/脚本同栈复用。  
5. **可协作的 AI 开发**：在仓库中固化 **`AGENTS.md` 与技能体系**，让编码代理在受控流程下可重复调用项目技能（见第 8 节）。

## 2. 产品目标

### 2.1 必须达成

- 提供 **只读** Web 浏览（写操作可后置独立 PRD）。  
- **Python 后端** 暴露稳定 API 与受控媒体访问。  
- **持久化数据层**（见第 5 节）：库、节点、资产、文件引用、（可选）标签与扫描任务状态等。  
- **关键路径**可观测、可配置、不裸奔路径。

### 2.2 非目标（V1 仍保持）

- 不承诺替代大型商业资产系统全能力。  
- 不承诺多租户 SaaS 级与无限扩展（另立项）。  
- 不承诺毫秒级实时目录同步（以扫描任务 + 定时/手动的 **同步水位** 表达）。

### 2.3 成功指标

| 指标 | 目标（建议） |
|------|----------------|
| 首屏可交互 | ≤ 2s（中等库、内网） |
| 列表首屏 | ≤ 1.5s，必须分页 |
| 可用性 | 单机/内网 7 天无计划外宕机 ≥ 99%（可定义维护窗口） |
| 可诊断性 | 库缺失/索引未就绪/媒体失败均有 API 错误码 + 日志 |

## 5. 数据架构（v2.0 重点）

### 5.1 双源模型：文件 vs 库表

- **文件侧**：现有 `CXA_Library` 布局保持不变（V1 以只读方式扫描）。  
- **数据侧（数据库）**：保存 **规范化** 的结构与元信息，供 API 与搜索使用。  
- **同步**：扫描服务将文件树投影为行记录，并记录 `sync_run_id`、**水位**、**异常列表**（单文件失败不拖垮全库）。

### 5.2 技术选型要求

- **主选**：**PostgreSQL** 或 **SQLite**（用 SQLAlchemy / Alembic 等，保证后续可升 PG）。  
- **禁止**把「业务真相」仅放在 JSON 文件或单次扫描的内存里（缓存可以，但非唯一）。  
- **迁移**：所有 schema 变更经迁移脚本；版本与部署文档同步更新。

### 5.3 逻辑实体（在 v1.0 表上强化）

| 实体 | 说明 |
|------|------|
| `libraries` | 库根、展示名、根路径、配置 JSON |
| `tree_nodes` | 层级节点（materialized path 或 `parent_id` + 序） |
| `assets` | 资产（叶子目录 → 条目的映射策略与现逻辑对齐） |
| `file_refs` | 主文件/缩略图/附属，含 `size`/`mtime`/`hash`（可选） |
| `tags` + `asset_tags` | 可选，便于产品化扩展 |
| `sync_runs` / `sync_errors` | 每轮扫描状态、可重试、可观测 |

**磁盘仍是文件真相；数据库是产品服务真相。** 二者通过同步任务与「校验状态」对齐。

### 5.4 与旧 `moudle_dict` 的边界

- 旧结构仅作为**参考输入**，不继续作为服务层对外的主模型。  
- 新代码中不再维护巨大嵌套 dict 为唯一状态；可保留极薄层「兼容读」至过渡期结束即删。

## 8. 与 `AGENTS.md` 及技能（OpenSkills）的协作

本仓库根目录的 **`AGENTS.md`** 用于声明 **对 AI 编码代理可用的技能**（与 OpenSkills 同步）。约定如下：

1. **技能发现**：`AGENTS.md` 中 `<available_skills>` 列出技能名与描述；与实施步骤相关的技能（如前端设计、安全审计、发布流程）在对应步骤启动前 **安装/更新** 并 **`openskills sync` 回写**（见仓库说明或贡献指南）。  
2. **使用方式**（对代理）：当任务与某技能描述匹配，通过 `npx openskills read <skill-name>` 加载 `SKILL.md` 的完整指令；多技能可逗号拼接。  
3. **与本项目的关系**：
   - **PRD/步骤文档** 定义「做什么、何时停」；**技能** 定义「如何做某类事」（例如 UI 标准、安全清单）。  
   - 分步开发时，**优先**在 `AGENTS.md` 中挂上与本步相关的技能，减少口头重复。  
4. **维护责任**：当技术栈或目录约定变更，更新技能正文或 `AGENTS.md` 的 `<available_skills>`，保持与真实现状一致。

> 注：本 PRD 不规定必须安装哪几个公网技能；由**每步开始前的轻量计划**与负责人共同确定，并写入该步的「输入/技能」小节。
